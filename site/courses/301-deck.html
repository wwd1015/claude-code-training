Module 1 · 40 min

Skill Creation I: Anatomy of a Skill

  • Explain what a skill is and when Claude Code loads one
  • Read and write the SKILL.md structure — frontmatter plus instructions
  • Write a skill description the matcher fires on — trigger phrases, not marketing
  • Decide when a task deserves a skill versus a one-off prompt
Frame skills as "writing the runbook once." Contrast with slash commands from CC 201: commands are text shortcuts you invoke by name; skills carry full workflows Claude can pick up automatically.
Module 1 · Anatomy of a Skill

What a skill is

  • A folder with a SKILL.md — reusable instructions Claude follows
  • Lives in .claude/skills/ (shared with the repo) or ~/.claude/skills/ (just you)
  • Loads automatically when your request matches its description
  • Write the runbook once instead of re-explaining it every session
The teammate analogy: you wouldn't re-teach the weekly report every week — you'd write it down. A skill is that write-up, in a form Claude discovers and executes.
Module 1 · Anatomy of a Skill

Two parts that matter

---
name: data-dictionary
description: >
  Profile a dataset and produce a standardized data dictionary.
  Use when asked to "profile this data", "document this dataset",
  or "make a data dictionary".
---

# data-dictionary
## Steps
1. Load the data; infer types, ranges, null rates per column.
2. Flag columns needing human description; never invent meanings.
...
Frontmatter drives discovery: name + description are ALL Claude sees before loading. The body loads only when triggered — progressive disclosure — so a hundred installed skills cost almost nothing until one is needed. Detail belongs in the body or linked files, never crammed into the description.
Module 1 · Anatomy of a Skill

The description is a trigger, not a tagline

  • The matcher only ever sees name + description
  • Bad: "A powerful helper for regression work" → never loads
  • Good: names what it produces + lists trigger phrases verbatim
  • Then "is my regression sound?" lands with no skill name typed
  • The body (the test battery) never loads during matching
Live demo idea: show the same skill failing on the vague description, then loading on the phrase-rich one — wording alone. Point at skills/regression-diagnostics/SKILL.md for the real "Use when the user says …" clause.
Module 1 · Anatomy of a Skill

Best practices & pitfalls

  • Write the description for the matcher; lead with the task, then "Use when …"
  • Keep SKILL.md lean; push heavy material to linked files
  • Name after the job (regression-diagnostics), not the tool
  • One skill, one workflow — split multi-purpose skills
  • Pitfall: detail crammed into the description bloats every match
  • Pitfall: a skill for a one-off task — wait for the second time you type it
Discipline split — For quants: first skill is usually a model-validation battery (VIF, Breusch-Pagan, ADF) whose steps must never drift. For data scientists: usually a profiling/triage pass over a large table; value is a consistent repeatable first look.
Lab · Module 1

Hands-on

  • Read skills/eda/SKILL.md — trigger phrases, drift-prone steps
  • Stub your own: frontmatter with your real trigger phrases
  • Stretch: say the task three ways; each should match the description
~10 min. This frontmatter becomes the skill they finish and test in module 2, so have them pick a genuine recurring task from their own work (CC 101 module 1).
Module 2 · 40 min

Skill Creation II: Design Patterns & Testing

  • Structure a skill as phases with hard rules so results don't drift
  • Make a skill self-contained enough to survive being shared
  • Test a skill the same way you'd test code — known input, expected output
Theme: the skill file IS the program. Treat it like code — structure it, then test it until it passes.
Module 2 · Design Patterns & Testing

Workflow-shaped skills: four patterns

  • Phases in order — numbered steps beat prose
  • Hard rules, stated as rules — "never do X / always do Y"
  • Output contracts — show the exact table/file format
  • Escape hatches — what to do when inputs are missing or ambiguous
These four are what stop drift. Point at skills/eda (phased profiling + output contract) and skills/backtest-report (hard rules about what may not be recomputed).
Module 2 · Design Patterns & Testing

Test it like code — four tests

  • Known input → expected output — run on a fixture, diff
  • Trigger test — ask in natural words; does it load?
  • Fresh-session test — /clear, rerun; no reliance on history
  • Break it once — odd input should fire the escape hatch
Self-contained matters: no machine-specific paths, discover locations ("look for a data/ directory"). Iterate on wording exactly as you would on failing code.
Module 2 · Design Patterns & Testing

Worked example: four patterns in one skill

  • skills/regression-diagnostics/SKILL.md shows all four
  • Phases: Fit → VIF → Breusch-Pagan → residuals → ADF → verdict
  • Hard rules: "report actual statistics, never guess"; "read-only on input"
  • Contract: names regression_report.md and each section's content
  • Escape hatch: assumes cross-section OLS unless told — asks if ambiguous
Walk the four tests against it live: known VIFs by hand and diff; "is my rates model sound?" to trigger; /clear to prove fresh-session; a single-column file to prove the escape hatch. All four pass → shippable.
Module 2 · Design Patterns & Testing

Best practices & pitfalls

  • Test on real, messy data before you share — not just the toy fixture
  • State hard rules as imperatives; show the output contract, don't describe it
  • Give every skill an escape hatch, or the model invents inputs
  • Push heavy reference material to a sibling file (progressive disclosure)
  • Pitfall: it works only because of earlier session context — catch with fresh-session
  • Pitfall: hard-coded absolute paths break on a teammate's clone
Discipline split — Quants: hard rules are statistical integrity ("report the actual test statistic", "state the significance level"). Data scientists: data hygiene and scale ("never mutate the raw partition", "sample before profiling a billion-row table").
Lab · Module 2

Hands-on

  • Finish the module-1 skill: phases, ≥2 hard rules, an output contract
  • Run the four tests against a small known input; ship to .claude/skills/
  • Stretch: fail the break-it test on purpose, then add the escape hatch
~15 min. The stretch — watching a confident wrong answer before adding the escape hatch — is what makes them write escape hatches for every skill afterward.
Module 3 · 45 min

Hooks: Guards & Automation

  • Explain what hooks are and where they fire in the agentic loop
  • Use a PreToolUse hook as a guard and a PostToolUse hook as automation
  • Write hooks with hygiene — least-scope matchers, fast scripts that fail loudly
  • Choose hooks over instructions when a rule must hold every time
The pivot of the course: everything before this is "asking nicely." A hook is the first deterministic guarantee — the harness runs it; the model can't skip it.
Module 3 · Guards & Automation

Deterministic enforcement

  • CLAUDE.md and skills are advisory — the model usually complies
  • A hook is a shell command the harness runs at fixed points
  • PreToolUse — runs before a tool call; can block it (guard rail)
  • PostToolUse — runs after; can't block, but can react (automation)
Other events exist (session start, prompt submit, on-stop) — mention they're in the docs. Hooks run with your permissions on your machine: read any hook before installing it.
Module 3 · Guards & Automation

Which mechanism? The decision rule

The rule is…Use
a preference ("prefer pandas idioms")CLAUDE.md
a workflow ("how we build reports")skill
a must-never / must-alwayshook
Two canonical patterns: Guard (PreToolUse) blocks writes to sensitive data — examples/hooks/guard-data.sh. Automation (PostToolUse) lints Python after each write — examples/hooks/run-checks.sh. Walkthrough in demo 05.
Module 3 · Guards & Automation

Worked example: data-guard wiring

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write|MultiEdit",
        "hooks": [
          { "type": "command",
            "command": "${CLAUDE_PROJECT_DIR}/.claude/hooks/guard-data.sh" }
        ]
      }
    ]
  }
}
The matcher is a regex over tool names, so the guard can't be dodged by using Write instead of Edit. The script reads the tool call as JSON on stdin and signals with its exit code: exit code 2 blocks and feeds stderr back to Claude; exit 0 allows. A minimal guard greps the target path for */data/raw/* and exits 2. Exact stdin schema + exit-code contract are in the hooks docs.
Module 3 · Guards & Automation

Best practices & pitfalls

  • Hook = must-ALWAYS-happen; CLAUDE.md = advisory
  • Keep scripts fast and fail loudly (they run on every matching call)
  • Scope the matcher to the tools that matter
  • Guard with PreToolUse; automate with PostToolUse — a post-hook is too late to block
  • Pitfall: a slow/flaky hook stalls the whole loop
  • Pitfall: over-broad matcher blocks legitimate writes (e.g. data/reports/)
Discipline split — Quants: protect a golden calibration input or validated model artifact from being overwritten mid-session. Data scientists: protect raw/immutable partitions and wire run-checks.sh to a linter for generated feature code.
Lab · Module 3

Hands-on

  • Install the sample guard hook; watch it block an edit to a protected file
  • Adapt it to a path that matters in your work (or wire run-checks.sh)
  • Stretch: instruct Claude to overwrite anyway — the hook still blocks
~15 min. The stretch is the whole lesson: the gap between the model agreeing to your instruction and the harness refusing the tool call. Have them commit the hook so a teammate inherits the guarantee.
Module 4 · 40 min

MCP I: Concepts & Connecting Servers

  • Explain what MCP is and what a server provides (tools, resources)
  • Connect a server via .mcp.json and choose the right scope
  • Decide when MCP beats a shell command for reaching a system
MCP is how Claude Code reaches systems beyond the filesystem — a purpose-built, permission-scoped interface instead of a general-purpose shell.
Module 4 · Concepts & Connecting

What MCP is

  • Open standard for plugging external systems into Claude Code
  • A server exposes a system as typed tools + readable resources
  • Claude Code is the client; you decide which servers it may talk to
  • A warehouse server exposes run_query + schema — not a raw shell
Without MCP, Claude reaches other systems by shelling out. With MCP it gets a scoped, legible interface — you can see exactly which tool ran and approve it by name.
Module 4 · Concepts & Connecting

Connecting a server + scope

{
  "mcpServers": {
    "warehouse": {
      "command": "npx",
      "args": ["-y", "some-sql-mcp-server", "--db", "analytics.duckdb"]
    }
  }
}
Project scope (.mcp.json in the repo) is shared with everyone who clones — right for team data sources. User scope follows you across projects — right for personal tooling. On first use Claude Code asks before trusting a project's servers.
Module 4 · Concepts & Connecting

Share safely: env expansion

"warehouse": {
  "command": "npx",
  "args": ["-y", "some-sql-mcp-server", "--dsn", "${WAREHOUSE_DSN}"],
  "env": { "WAREHOUSE_TOKEN": "${WAREHOUSE_TOKEN}" }
}
Check .mcp.json in so clones inherit the connection — but never a secret. ${VAR} is read from each person's environment at launch: the repo commits the shape, never the credential. Point the DSN at a read-only user (module 5).
Module 4 · Concepts & Connecting

MCP or a shell command? + practices

  • Local file you'd read once → shell
  • Recurring, structured, authed system → MCP server
  • Least privilege: hand a data server a read-only user
  • Keep the server list short, named per system
  • Pitfall: a token pasted into .mcp.json then committed → rotate + use ${VAR}
  • Pitfall: assuming a project server is trusted automatically (expect the prompt)
Discipline split — Quants: MCP fronts a market/reference DB or pricing service over curated views. Data scientists: a warehouse or feature store with very large tables, which makes module 5's result-limit and scoping defaults non-optional.
Lab · Module 4

Hands-on

  • Connect the DuckDB demo in examples/mcp/
  • Confirm: tools listed, a question routes through a tool call, removing the entry removes it
  • Stretch: move the connection string to ${VAR}; unset it and watch it fail to launch
~10 min. The stretch's failure-to-launch is the proof the secret was never in the file. Watch the permission prompt name the tool during the routed question.
Module 5 · 40 min

MCP II: Your Data over MCP

  • Run the schema-first querying pattern against a database over MCP
  • Apply the safety defaults: read-only credentials, scoped access, result limits
  • Scope an MCP integration for one data source your team actually uses
This is the payoff module: "query the warehouse" becomes a conversation, pointed at your data with the same agentic loop from CC 101.
Module 5 · Your Data over MCP

The schema-first pattern

  • Orient — Claude pulls schema before writing any SQL
  • Query in plain language — review the SQL it shows you
  • Iterate on results — one query's result informs the next
  • Land the artifact — save the query + output as files in the repo
Walkthrough is demo 04 (SQL over MCP), runnable setup in examples/mcp/. The landed artifact is what makes the session reproducible and reviewable — not a number that scrolled past in chat.
Module 5 · Your Data over MCP

Safety defaults (non-negotiable)

  • Read-only credentials — the DB user simply cannot write
  • Scope the surface — expose the views the team needs, not the whole warehouse
  • Limit result sizes — a SELECT * should truncate, not flood context
  • Your data-handling policy still applies — the model sees returned rows
Tie back to CC 201: context is finite. Views are also where row-level policy and column masking live, so pointing the server at the view layer carries policy into every query.
Module 5 · Your Data over MCP

Worked example: scope before you connect

-- Read, and only read, the curated views.
CREATE ROLE claude_ro NOLOGIN;
GRANT USAGE  ON SCHEMA analytics_curated TO claude_ro;
GRANT SELECT ON ALL TABLES IN SCHEMA analytics_curated TO claude_ro;

-- Login user the MCP server uses, holding only that role.
CREATE USER claude_mcp LOGIN PASSWORD :'from_secret_manager';
GRANT claude_ro TO claude_mcp;
Least privilege isn't a Claude Code setting — it's how you provision the DB user the server logs in as. Point the DSN at claude_mcp, expose only analytics_curated, set a result cap. The four defaults are now enforced by the credential, not by memory.
Module 5 · Your Data over MCP

Best practices & pitfalls

  • Provision least privilege in the database, not in the prompt
  • Expose views, not base tables; cap result size and prefer aggregates
  • Land a reviewable artifact (a saved .sql + output) every time
  • Pitfall: connecting with your own read-write account "just to test"
  • Pitfall: pointing the server at raw schemas → PII into the transcript
  • Pitfall: no result cap, then a SELECT * on a fact table
Discipline split — Quants: curated views are reference/market data + desk P&L; land the query next to the model that consumes it. Data scientists: tables are large and wide, so "aggregate, don't dump" and the result cap matter most.
Lab · Module 5

Hands-on

  • Run the four-step pattern end to end; land a saved query + output file
  • Write a five-line scoping memo for the real data source
  • Stretch: write the actual CREATE ROLE / GRANT SELECT and hand it to the warehouse owner
~15 min. The stretch turns the memo into a reviewable request a data-platform team can actually approve — a scoped, read-only integration, not "give the agent my account."
Module 6 · 35 min

Packaging & Sharing What You Build

  • Share skills, commands, and hooks with your team through the project repo
  • Describe what plugins add on top of repo-level sharing
  • Use the contributed-library path to give your work a life beyond your team
Everything built in this course ships by committing it. The repo is the distribution channel — and the training material.
Module 6 · Packaging & Sharing

The repo is the distribution channel

  • .claude/skills/ — skills, picked up by everyone who clones
  • .claude/commands/ — slash commands (CC 201)
  • settings.json + hook scripts — guards and automation
  • .mcp.json — team data sources (credentials via env, never committed)
  • A teammate clones and has the whole toolkit — no install step
Pre-commit checklist: passes the four tests from module 2; no machine-specific paths; secrets via environment; one README line saying what it's for.
Module 6 · Packaging & Sharing

Plugins & the contributed library

  • Plugins bundle skills/commands/hooks/MCP for distribution across repos
  • Reach for them only when the same toolkit must land in many repos
  • Seed examples are program-maintained references
  • Contributed entries are team-authored, author-owned, promoted from real work
  • Stale Contributed entries are archived, not silently fixed
Promotion path: prove it in your own repo → tag with discipline/use-case front-matter → submit per CONTRIBUTING.md. A Champion can fold a team-specific skill into the LOB overlay instead.
Module 6 · Packaging & Sharing

Promoting a skill from "mine" to "ours"

  • Re-run the four tests on a clean clone (catches "works on my machine")
  • Strip machine-specifics; confirm no baked-in credential or hostname
  • Add the one-line README + discipline tags
  • Decide destination: Contributed library vs LOB overlay — this is the exercise
  • Commit the whole toolkit — guard hook and .mcp.json too
Broadly useful → Contributed entry that keeps your name on it. Encodes something only your desk does → hand to your Champion for the LOB overlay. Same skill, narrower home.
Module 6 · Packaging & Sharing

Best practices & pitfalls

  • Ship by committing, not by sending config around
  • Prefer project scope for anything a second person uses
  • Never commit a secret; use env expansion for .mcp.json
  • Gate every shared skill on the four tests + a clean-clone check
  • Pitfall: personal scope for a team skill — nobody who clones gets it
  • Pitfall: a plugin/marketplace when one committed folder would do
Discipline split — Quants: Contributed entries are often desk-specific econometric conventions → frequently the LOB overlay. Data scientists: reusable EDA/triage/feature-QA that generalizes → the shared Contributed library.
Lab · Module 6

Hands-on

  • Make your module-2 skill shareable: checklist, README line, open the submission
  • If team-specific, hand it to your champion for the LOB overlay
  • Stretch: clone into a fresh directory and run it with zero setup; fix every break
~10 min. The clean-clone run is the real bar for "shareable" — whatever breaks there is the actual pre-commit checklist, not the idealized one.

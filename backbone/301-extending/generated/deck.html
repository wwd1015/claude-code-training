Module 1 · 30 min

Skill Creation I: Anatomy of a Skill

  • Explain what a skill is and when Claude Code loads one
  • Read and write the SKILL.md structure
  • Decide when a task deserves a skill vs a one-off prompt
Framing: everyone has re-explained the same workflow to Claude five times. A skill is writing the runbook once.
Module 1 · Anatomy of a Skill

A skill is a runbook Claude picks up itself

  • A folder with a SKILL.md — reusable workflow instructions
  • .claude/skills/<name>/ in a repo → shared with the team
  • ~/.claude/skills/<name>/ → personal, follows you
  • Loaded automatically when your request matches its description
Contrast with re-prompting: telling a new teammate how to run the weekly report every week vs writing the runbook once. Project-level skills ship with the repo — clone = installed.
Module 1 · Anatomy of a Skill

Two parts, two jobs

---
name: data-dictionary
description: >
  Profile a dataset and produce a standardized data dictionary.
  Use when asked to "profile this data" or "make a data dictionary".
---
## Steps
1. Load the data; infer types, ranges, null rates per column.
2. Flag columns needing human description; never invent meanings.
3. Write dictionary.md using the table format below.
Frontmatter drives DISCOVERY — name + description are all Claude sees before deciding to load. Description must contain the phrases people actually say. Body loads only when triggered = progressive disclosure: 100 installed skills cost ~nothing until needed. Detail goes in the body, never crammed into the description.
Module 1 · Anatomy of a Skill

Skill, or just a prompt?

  • Typed the same instructions twice → skill
  • Steps or rules that must not drift → skill
  • Teammates need identical behavior → skill
  • Genuinely one-off → just prompt
  • Text shortcut you invoke by name → slash command (CC 201)
Key distinction vs slash commands: commands are invoked by name; skills can trigger automatically on matching requests and carry full workflows.
Lab · Module 1

Module 2 · 30 min

Skill Creation II: Design Patterns & Testing

  • Structure a skill as phases with hard rules so results don't drift
  • Make a skill self-contained enough to survive sharing
  • Test a skill like code — known input, expected output
Theme: the skill file IS a program; treat it with code discipline.
Module 2 · Patterns & Testing

Durable skills read like runbooks

  • Phases in order — numbered steps beat prose
  • Hard rules — "never modify raw data files"
  • Output contracts — show the exact format; example beats description
  • Escape hatches — missing input? "ask one question, then proceed"
Worked examples to open live: skills/eda (phased profiling + output contract), skills/backtest-report (hard rules about what may not be recomputed). Without escape hatches, the skill guesses confidently on bad input.
Module 2 · Patterns & Testing

The four tests

  • Known input → expected output; diff the result
  • Trigger test — your natural words load it (not its name)
  • Fresh session — /clear, run again; no conversation dependence
  • Break it once — bad input should hit the escape hatch
Also: self-contained or it breaks — no machine-specific paths, dependencies declared at top. Iterate on wording exactly like failing code. Trigger-test failures = description missing the user's phrasing, fix the frontmatter not the body.
Lab · Module 2

Hands-on: finish and ship it

  • Finish module 1's stub: phases + 2 hard rules + output contract
  • Run all four tests on a small known input
  • Ship to .claude/skills/ in one of your real repos
~15 min. This is the course's central artifact — everything later (sharing, LOB overlay, contributed library) builds on this skill existing. Circulate and check output contracts: most people write descriptions instead of examples.
Module 3 · 30 min

Hooks: Guards & Automation

  • Explain what hooks are and where they fire in the loop
  • Use PreToolUse as a guard, PostToolUse as automation
  • Choose hooks over instructions when a rule must hold every time
The one-liner for this module: skills ask nicely, hooks enforce.
Module 3 · Hooks

Asking nicely vs enforcement

  • CLAUDE.md & skills = instructions the model almost always follows
  • A hook = a shell command the harness itself runs
  • Can't be skipped, forgotten, or talked around
  • PreToolUse — inspects a tool call, can block it
  • PostToolUse — reacts after success; can't block
Configured in settings.json against lifecycle events. Others exist (session start, prompt submit, on-stop) — point to docs. Emphasize: deterministic, fires every time regardless of conversation state.
Module 3 · Hooks

Two canonical patterns, both in the repo

  • Guard: examples/hooks/guard-data.sh — blocks writes to protected data paths
  • Automation: examples/hooks/run-checks.sh — lints after every edit; Claude fixes its own output
  • Wiring: examples/hooks/settings.sample.json — copy, adapt paths, done
Demo live if time: install guard in a scratch repo, ask Claude to edit a protected file, watch the block. Walkthrough is demos/05-hooks-automation.md. Safety note: hooks run with YOUR permissions — read hook scripts before installing, like any code you didn't write.
Module 3 · Hooks

The decision rule

  • Preference ("prefer pandas idioms") → CLAUDE.md
  • Workflow ("how we build reports") → skill
  • Must-never / must-always → hook
Quiz the room with examples: "never write to prod_configs/" (hook), "our plots use this style" (CLAUDE.md), "monthly risk memo format" (skill). This table is the module's take-home.
Lab · Module 3

Hands-on: make a rule unbreakable

  • Install the sample guard hook in a scratch repo
  • Ask Claude to edit a protected file — watch it block
  • Adapt the guard to a path that matters in your work
  • (or) wire run-checks.sh to your linter; confirm it fires
~12 min. Common trip-up: hook script not executable (chmod +x), or settings path wrong. The "watch it block" moment is the payoff — don't let anyone skip it.
Module 4 · 30 min

MCP I: Concepts & Connecting Servers

  • Explain what MCP is and what a server provides
  • Connect a server via .mcp.json and choose the right scope
  • Decide when MCP beats a shell command
Shift of scene: skills/hooks extended Claude's behavior; MCP extends its REACH.
Module 4 · MCP Concepts

MCP: plug external systems in, on your terms

  • Open standard; Claude Code is the client
  • A server exposes a system as typed tools + resources
  • Warehouse server = run_query + schema catalog, not a shell
  • You decide which servers it may talk to
Contrast with the status quo: without MCP, Claude reaches systems via shell commands. MCP gives a purpose-built, permission-scoped interface. Tool calls flow through the same permission model from CC 101.
Module 4 · MCP Concepts

Connecting: .mcp.json + scope

{
  "mcpServers": {
    "warehouse": {
      "command": "npx",
      "args": ["-y", "some-sql-mcp-server", "--db", "analytics.duckdb"]
    }
  }
}
  • Project scope (in repo) → team data sources, ships with clone
  • User scope → personal tooling, follows you
Also: claude mcp add. First use of a project's servers prompts for trust. Auth: OAuth or env-var token — NEVER hard-code credentials in .mcp.json. Org-approved server list is an LOB/verify item.
Module 4 · MCP Concepts

MCP or shell?

  • One-off on a local file → shell
  • Recurring, structured ("query the warehouse") → MCP
  • Needs auth / scoping / an API → MCP
  • Rule of thumb: would you write a wrapper library? → wrap it as MCP
Keep this pragmatic — MCP is not a religion; a trusted local script via shell is fine. The wrapper-library heuristic lands well with this audience.
Lab · Module 4

Module 5 · 30 min

MCP II: Your Data over MCP

  • Run the schema-first querying pattern over MCP
  • Apply safety defaults: read-only, scoped, result-limited
  • Scope an MCP integration for a data source your team uses
This is the module that changes daily work for a data audience — budget demo time accordingly.
Module 5 · Data over MCP

The schema-first pattern

  • 1 Orient — "what tables exist? what's in trades?"
  • 2 Query in plain language — review the SQL it shows
  • 3 Iterate — each result informs the next question
  • 4 Land the artifact — saved query + output file in the repo
Live demo: demos/04-sql-over-mcp.md against the DuckDB sandbox. It's the CC 101 agentic loop pointed at data instead of files. Step 4 is what makes sessions reproducible — the end product is a reviewable, rerunnable file, not a chat transcript.
Module 5 · Data over MCP

Safety defaults — non-negotiable

  • Read-only credentials — agent mistakes get a ceiling
  • Scope the surface — needed schemas/views, not the warehouse
  • Limit result sizes — truncate, don't flood the context
  • Data-handling policy still applies: the model sees returned rows
Views are where row-level policy lives. Result limits tie back to CC 201's context lesson. Org policy link + approved endpoints are verify/LOB items — tell the room who their champion is.
Lab · Module 5

Hands-on: end to end, then scope the real thing

  • Write a 5-line scoping memo: source · schemas/views · who issues the read-only credential · first 3 questions
~15 min. The scoping memo is the bridge to real adoption — collect them; they're the pipeline of future MCP integrations (and 501 capstone candidates).

Module 6 · Packaging & Sharing

The repo is the distribution channel

  • .claude/skills/ · .claude/commands/ · settings.json + hooks · .mcp.json
  • Clone = installed; no setup step
  • Credentials via env — never committed
  • Plugins: bundles for many unrelated repos — know they exist
Pre-commit checklist for a shared skill: four tests pass, no machine-specific paths, secrets via env, one README line. Plugins/marketplaces: verify current mechanism against the skills docs before teaching in depth.
Module 6 · Packaging & Sharing

Seed vs Contributed — how the library grows

  • Seed — program-maintained references (this repo's skills/, demos/)
  • Contributed — team-authored, author-owned, promoted from real work
  • Stale → archived, not silently fixed
  • Path: prove it → tag it → submit per CONTRIBUTING.md
  • Team-specific? → your champion's LOB edition instead
This is the governance model (ADR 0001): the real library is contributed, capstones (CC 501) are the main promotion source. Deciding backbone-vs-LOB is itself the skill to teach here.

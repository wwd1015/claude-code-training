Module 1 · 35 min

Subagents: Delegation & Decomposition

  • What a subagent shares with your session — and what it doesn't
  • Define a reusable subagent in .claude/agents/
  • Decide: delegate vs. do it inline
Ask who has noticed Claude "spawning a task" during search — most have seen ad-hoc subagents without knowing it.
Module 1 · Subagents

A separate Claude, on a scoped task

  • Own context window · own system prompt · own (restricted) tools
  • Doesn't see your conversation — only its task prompt
  • You see only its final report, not its tool-by-tool work
  • Context isolation: it reads 50 files, you keep 1 conclusion
  • Blind spots: everything it needs must be in the brief or on disk
The isolation is the feature AND the failure mode. Story to tell: a subagent asked to "fix the bug we discussed" — it never saw the discussion. Brief like you're handing off to a colleague who just walked in.
Module 1 · Subagents

Defining a reusable role

  • .claude/agents/<name>.md — front-matter: name, description, allowed tools
  • Body = its system prompt (role, rules, output contract)
  • Project agents ship with the repo; personal ones in ~/.claude/agents/
  • Worked example: demo 03 — anomaly-triage agent (read-only, structured report)
Open demos/03-anomaly-triage-agent.md live if time allows. Emphasize the description field — it's how the main agent decides when to use this role, so write it as "use when…".
Module 1 · Subagents

The punchline rule: the cost of delegation is the briefing. If writing the prompt takes longer than the task, do it inline.

Module 2 · 35 min

Designing Multi-Agent Workflows

  • The four core workflow shapes
  • What fans out vs. what stays sequential
  • Designing for failure
This is the design module — no new syntax, only architecture. Whiteboard-heavy.
Module 2 · Workflow design

The four shapes

  • Fan-out / map — same task, many items · wall-clock ≈ slowest item
  • Pipeline — items flow through stages, no barrier between items
  • Generate-then-verify — independent checker, adversarial framing
  • Orchestrator + workers — your session already is one
On verify: the checker gets ONLY the artifact + acceptance criteria, never the generator's reasoning — otherwise it inherits the blind spots. "Try to refute this" beats "check this". Draw each shape as boxes/arrows on the whiteboard.
Module 2 · Workflow design

What fans out?

  • Fan out: items independent + self-contained to brief
  • Sequential: outputs chain · shared-resource conflicts · discovery reshapes the plan
  • Method: list the work → draw "needs the output of" arrows
  • Arrow-free groups fan out; chains become pipeline stages
Do a live example with a deliverable the room actually produces (ask for one — periodic report, multi-dataset QA). Discovery-before-fanout is the subtle one: if step 1 changes what steps 2–10 should be, it can't be inside the fan-out.
Module 2 · Workflow design

Design for failure

  • Empty result — "no findings" or "task failed"? Force NONE, never ambiguous silence
  • Wrong result — verify stage (systematic) or spot-check (cheap); match rigor to blast radius
  • Partial completion — synthesize what arrived and name what's missing
The dangerous report is the one that covers 7 of 10 items and reads as complete. Make "name what's missing" a hard requirement of every synthesis prompt.

Module 3 · 30 min

Headless & Automation

  • Run Claude Code non-interactively with claude -p
  • Place agents in CI and scheduled jobs
  • The unwatched-agent discipline
Turning point of the course: the human leaves the loop here.
Module 3 · Headless

Print mode: Claude as a Unix tool

claude -p "summarize what changed in this repo since yesterday
          and flag anything touching the pricing model" > daily-brief.md
  • One task, no session, result to stdout
  • --output-format json + piped input → composable
  • cron · Makefiles · shell scripts · notebook pre-flight checks
Run it live against the training repo if you have credentials on the demo box. CLI reference covers flags: permission mode, allowed tools, model, output format.
Module 3 · Headless

Where automation fits

  • Scheduled chores — nightly data-drop QA + triage note
  • CI steps — PR diff review against a checklist; regenerate stale docs
  • Event-driven — pipeline failure → agent gathers logs, drafts the incident summary
GitHub Actions can run Claude Code as a workflow step — link the org's approved setup here (verify marker in the module). Ask the room: what do you check by hand every morning? Those are the candidates.
Module 3 · Headless

The unwatched-agent discipline

  • Verification is part of the task — and the job fails loudly when the check fails
  • Permissions pinned down — explicit allowlists, narrowest mode that works
  • Output lands where a human looks — PR comment, dashboard, alert channel
The failure mode to burn in: the agent that "declares victory". An unwatched agent without a verification step is worse than no agent — it manufactures false confidence on a schedule.

Module 4 · 30 min

The Agent SDK: Building Custom Agents

  • What the SDK provides vs. Claude Code
  • Three design surfaces: prompt, tools, permissions
  • Choosing the right layer
Expectation-setting: this is a design module, not a coding tutorial — the SDK course + docs carry the code.
Module 4 · Agent SDK

The same engine, programmable

  • The loop, tools, permissions, context management — as a library
  • Your application becomes the harness (not a person in a terminal)
  • It decides: what the agent works on · which tools exist · where results go
Everything taught so far transfers: skills thinking → system prompts, allowlist thinking → permission hooks. That's the reassuring message — the SDK is the same concepts with code around them.
Module 4 · Agent SDK

Three design surfaces

  • System prompt — role, rules, output contract (your CC 301 skill-writing, verbatim)
  • Tools — built-ins, MCP servers, custom tools; the list IS the capability boundary
  • Permissions — auto / confirm / forbid, in code (your CC 201 allowlists)
Drill: for any proposed agent, ask the three questions in order — what does it know, what can it touch, what must it ask about.
Module 4 · Agent SDK

Pick the lowest layer that fits

The anti-pattern to name: reaching for the SDK when a skill + cron'd claude -p would do — you inherit deployment, monitoring, and upgrades for zero added capability. SDK is right when the agent must live where Claude Code isn't.

Module 5 · 35 min

Best Practices: Context, Cost & Evaluation

  • Context engineering — what enters the window decides quality
  • Cost drivers and the standard controls
  • A lightweight evaluation loop
The "operate it well" module. Everything here is cheap to adopt and compounds.
Module 5 · Best practices

Context engineering

  • Quality = f(context window): every file read, every output, every turn
  • Feed conclusions, not raw material — that's what subagents are for
  • CLAUDE.md is standing context — short and current; stale lines are noise
  • New task → new context: /clear beats coaxing a drifted session
  • Contracts in writing: output format, definition of done, verification
Highest-leverage slide of the module. Anecdote format works: same prompt, clean vs. polluted context, different answers.
Module 5 · Best practices

Cost: drivers → controls

  • Drivers: context size · agent count · iteration loops · model choice
  • Match model to step — cheap for mechanical, strongest for judgment
  • Cap fan-outs and retries in the design — unbounded loops are cost incidents
  • Re-reading the same big files? Build a summary artifact, not a bigger budget
Model-per-subagent is a design decision most people don't know they have. The re-reading pattern is the most common silent burn in practice.
Module 5 · Best practices

Evaluation: three tiers

  • Spot-check protocol — fixed sample rule, not ad-hoc glancing
  • Golden tasks — known-good suite; run on every prompt/skill/model change
  • Regression prompts — every production failure becomes a golden task
  • Track ONE adoption metric from day one (minutes saved per run)
"Golden tasks are the unit tests of your agent system" is the phrase that sticks. The metric point matters organizationally — it's what justifies or kills the system at review time.
Lab · Module 5

Hands-on: evaluate your automation

  • Three golden tasks for your 401-03 automation, expected outputs written down
  • Run · record in evals.md
  • "Improve" the prompt → rerun → golden tasks arbitrate
  • Estimate monthly token cost; name the biggest driver
~15 min. The arbitration step is the lesson: several "obvious improvements" will fail their golden tasks, and that surprise is the point.
Module 6 · 25 min

Governance & Safety for Agent Systems

  • Least privilege for agent permissions and credentials
  • Observable, auditable agent actions
  • Review gates before irreversible actions
Close the day on trust: this module is what makes the previous five acceptable to risk/compliance.
Module 6 · Governance

Least privilege, at scale

  • Permissions per role, not per person — reviewer reads, doc-gen writes docs/ only
  • Credentials scoped to the job — read-only service accounts, never personal creds
  • Blast radius by placement — sandboxes that can't reach production paths
Personal credentials are the hill to die on: an unattended agent on your creds does everything you can do and audits as YOU. Insert the org's service-account provisioning process here (verify marker in module).
Module 6 · Governance

Observability & audit

  • Keep the artifacts — reports, diffs, verification evidence, like build logs
  • Changes through git — free audit log with rollback included
  • Log the inputs — version prompts/skills like code, because they are
The test: can you answer "what did the agent do, and why?" a week later from artifacts alone? The lab checks exactly this.
Module 6 · Governance

Review gates: the reversibility test

  • Reversible (branch edit) → no gate needed
  • Irreversible (merge to main, external messages, shared DB writes, deletes) → human gate
  • The scaling pattern: agents produce artifacts, humans promote them
PRs and drafts are the canonical artifact-promotion mechanisms — agent speed on the safe side of the line, human judgment exactly where it pays.
Lab · Module 6

Hands-on: audit your automation

  • Write its permission set + credential scope — tighten at least one thing
  • Reconstruct its last run from artifacts alone
  • Mark each action: reversible or gate-required
  • Fix the gaps
~10 min. Everyone finds at least one gap — usually output landing nowhere or a too-broad approval. That's the intended takeaway.
